使用Py_BEGIN_ALLOW_THREADS - 时，如何避免在Python的C扩展的海湾合作​​委员会警告

The simplest way to manipulate the GIL in Python C extensions is to use the macros provided:操纵GIL在Python的C扩展，最简单的方法是使用提供的宏： 

my_awesome_C_function() { blah; Py_BEGIN_ALLOW_THREADS // do stuff that doesn't need the GIL if (should_i_call_back) { Py_BLOCK_THREADS // do stuff that needs the GIL Py_UNBLOCK_THREADS } Py_END_ALLOW_THREADS return blah blah; }

This works great, letting me release the GIL for the bulk of my code, but re-grabbing it for small bits of code that need it.这个伟大的工程，让我释放GIL的大部分，我的代码，但需要少量的代码重新抓住它。

The problem is when I compile this with gcc, I get:问题是，当我用gcc编译这个，我得到： 

 ext/engine.c:548: warning: '_save' might be used uninitialized in this function

because Py_BEGIN_ALLOW_THREADS is defined like this:因为Py_BEGIN_ALLOW_THREADS定义是这样的： 

 #define Py_BEGIN_ALLOW_THREADS { \ PyThreadState *_save; \ _save = PyEval_SaveThread();

So, three questions:因此，以下三个问题：

  1. Is it possible to suppress gcc's warning,它可以抑制gcc的警告，
  2. Does anyone have any idea why gcc thinks _save might be used uninitialized, since it is assigned to immediately after its declaration, and有没有人有任何想法，为什么海湾合作委员会认为_save可能会使用未初始化的，因为它是分配给后立即声明，并
  3. Why wouldn't the macro have been defined to declare and initialize the variable in one statement to avoid the issue?为什么不宏被定义来声明和初始化的变量在一条语句中，以避免该问题吗？

(the last two are really just for my own curiosity). （最后两个其实只是我自己的好奇心）。

I can avoid the issue by not using the macros and doing it all myself, but I'd rather not.我能避免不使用宏和做的一切都是我自己的问题，但我宁可不要。

来源：http://stackoverflow.com/questions/2147029/how-to-avoid-gcc-warning-in-python-c-extension-when-using-py-begin-allow-threads
    1. Yes, it is possible to suppress uninitialized warnings using the -Wno- prefix.是的，这是可能的以抑制未初始化警告使用-Wno前缀。

    -Wall -Wno-uninitialized

    If you want to remove just this warning, you could simply initialize _save to a null pointer so that it doesn't rely on a function return value...如果你想删除此警告，你可以简单地初始化为空指针_save因此，它不依赖于一个函数的返回值... that one line of code and a comment makes sense to me:这一行的代码和评论对我来说很有意义： 

     PyThreadState *_save; _save = 0; /* init as null pointer value */ _save = PyEval_SaveThread();
