DBus

1. AddMatchRule indicates which messages ( signals ) the client is caring about.
2. Use dbus_connection_add_filter to add a filter function to a dbus connection. The filter is called prior to calling the object handing function.

3. When gvfsd receives a calling of 'mount_location', it will spawn a backend process (spawn_mount) with the parameters dbus_id and object_path. Then the child process can call back the method with the dbus_id and object path.

/usr/lib/gvfsd-ftp --spawner :1.7 /org/gvfs/mountable/spawnd

4. gvfsd reads all mountable configuration from /usr/share/gvfs/mounts and registers them in a list 'mountables'
gvfsd registers its objects in mount_init
the service function is dbus_message_function
the filter function is mount_tracker_filter_func which only handles 'NameOwnerChanged' functions.

5. gvfsd and gvfsd-xxx has similar architecture.
daemon_start_mount will be called in daemon_message_func which is a daemon's filter function. Here, a backend will be created.
l register the dbus handler 'backend_dbus_handler' where most method calls are handled.
